# 第二阶段：从原始数据到可分析数据

> 适用对象：已经理解 Phase 1，但还不明白“为什么要清洗、扁平化、重新编码”的学生  
> 学习目标：理解 C04-C13 在整个研究中的作用，知道一份统计分析数据集是怎样被制造出来的

---

## 1. 第二阶段在解决什么问题

经过第一阶段，我们知道数据“能读、能合并、结构大体可信”。  
但这并不代表它已经适合直接做统计。

原始数据的问题在于：

- 字段嵌套很深
- 有些列对分析没有帮助
- 有些列看似有用，但定义口径不一致
- 有些任务标签要重新整理才能拿来做分层分析

所以第二阶段的目标是：

**把复杂原始对象加工成“分析友好型数据表”。**

---

## 2. C04-C11 为什么要逐字段“摸底”

这一组 `touch_` 脚本看起来很碎，但教学价值很高。  
它们像医生问诊前的体检，不是为了直接下结论，而是为了回答：

**这个数据到底长什么样？每个字段能不能信？能不能用？**

### 2.1 C04：看 Session 分布
- 让学生知道会话数有多少
- 知道大多数会话只包含 1 个 Order 还是多个 Order

### 2.2 C05：看 Order 分布
- 让学生看清 `evaluation_order = 1` 占绝大多数
- 为后面“只保留首轮评价”埋下伏笔

### 2.3 C06：看 Turn 分布
- 解释一条记录内部对话有多少轮
- 让学生理解“回答长度”并不只是单句长短，而和对话轮次有关

### 2.4 C07：看模型参与情况
- 哪些模型出现得更多
- 哪些模型胜率更高
- 为后面“模型能力可能是混淆变量”打基础

### 2.5 C08-C09：看角色与内容结构
- 确认 `conversation_a/b` 主要是 `user/assistant` 两类角色
- 判断哪些嵌套字段可以扁平化、哪些属于冗余信息

### 2.6 C10：验证 token 字段口径

这是整个项目中非常关键的一步。  
它直接发现了一个“如果不检查就会把后面分析全部带偏”的问题：

- 代码会把对话内部各条消息的 `num_tokens` 与 `conv_metadata` 中的汇总字段逐项对照
- `sum_assistant_a_tokens` 和 `sum_assistant_b_tokens` 是当前评价对象的回答长度
- 但 `sum_user_tokens` 在 `evaluation_order > 1` 时会累积整个 Session 历史

这意味着：

**如果直接把后续轮次保留下来，用户输入长度就不再只属于当前这次评价，而掺进了更早轮次的上下文。**

课堂上可以把这个问题讲成：

**“我们以为在比较这一题，结果里面偷偷混进了上一题和上上题的字数。”**

### 2.7 C11：看 category_tag
- 把标签体系拆开看
- 明白哪些标签可以转成布尔变量
- 为后面子集划分准备材料

### 2.8 这一组脚本虽然不直接建模，却决定“哪些字段可以进模”

这是第二阶段最容易被学生低估的一点。  
C04-C11 的作用不是产出显著性结果，而是给后面的 C12 一个决策依据：

- 哪些字段只是原始结构，需要扁平化后再用
- 哪些字段存在口径污染，必须过滤后才能解释
- 哪些标签能安全地变成布尔变量和质量维度列

如果少了这一步，后面的建模就会变成“先分析，再回头解释字段”，研究顺序是倒过来的。

## 3. C12 为什么是整个工程的转折点

C12 不是简单的清洗脚本，它是从“原始数据世界”进入“分析数据世界”的分界线。

### 3.1 C12 做了什么

它主要做了五件事：

1. 过滤不适合分析的记录
2. 把对话、元数据和标签重组为更稳定的 nested schema
3. 把长度、格式和任务标签的“原料字段”统一收进 `metadata_*` 和 `category_tag`
4. 保留后续分析真正需要的最小顶层结构
5. 写出统一的优化数据表和清洗报告

### 3.2 最关键的过滤规则：只保留 `evaluation_order = 1`

这是本项目最值得反复强调的规则之一。  
原因不是“首轮更重要”，而是“后续轮次的用户 token 口径被污染”。

所以保留首轮的逻辑是：

- 不是随意裁剪样本
- 而是为了保证长度变量的解释一致
- 让后续所有统计都对应“同一层级、同一口径”的比较

### 3.3 C12 还过滤了什么

- `conversation_a` 或 `conversation_b` 中空内容
- `language = <err>` 的异常语言识别行
- 关键评分字段为 `None` 的异常行

再细一点说，代码里排除的不只是“空字符串”，而是对话段里 `content` 为空列表或空 `ndarray` 的情况。  
这类样本如果不先处理，后面扁平化时会留下“看似存在、其实没有内容”的伪文本列。

这些过滤的意义可以简单告诉学生：

**分析不是把所有数据都硬塞进去，而是只保留那些“定义清楚、口径一致、能支撑解释”的数据。**

## 4. 结构重组到底是什么意思

原始数据里的对话、token 汇总和标签本来就是多层嵌套对象。  
当前版本的 C12 不再把所有信息一口气摊平成几十个顶层列，而是把它们重组为更稳定的分析底座。

### 4.1 重组前
- 一条记录里包含多层字典和列表
- 对话内容、token 汇总、格式计数、任务标签都散落在不同字段里
- 如果直接把所有内容摊平成很多列，后续 schema 会不断膨胀

### 4.2 重组后
- `conv_a` / `conv_b` / `conv_user`：保留 A 侧、B 侧和用户侧的对话结构
- `metadata_a` / `metadata_b` / `metadata_user`：统一承接长度与格式计数原料
- `category_tag`：统一承接四类任务布尔值和 7 个质量维度

这里最值得讲给学生听的一点是：  
**分析数据结构不一定等于“尽量多列”，而是要等于“最容易稳定复用的结构”。**

当前项目采用的是“持久层保持紧凑、分析层按需展开”的思路：先把 `optimized_data.parquet` 写成较小、较稳定的 nested schema，再由后续脚本在分析阶段把所需变量展开出来。

## 5. 特征工程：有些变量持久化，有些变量在分析阶段再展开

这一阶段最适合顺便讲“操作化”这个概念。

### 5.1 持久层里保存的“原料字段”
- 长度原料保存在 `metadata_a`、`metadata_b`、`metadata_user`
- 格式原料也保存在 `metadata_*` 里的标题、列表、粗体计数
- 任务类别与质量维度统一收进 `category_tag`

### 5.2 分析层再展开的变量
- `a_tokens`、`b_tokens`、`user_tokens`
- `a_header_count`、`a_list_count`、`a_bold_count`
- `b_header_count`、`b_list_count`、`b_bold_count`
- `creative_writing_bool`、`if_bool`、`math_bool`、`code_bool`

这些变量并不都直接持久化在 `optimized_data.parquet` 顶层。  
当前版本里，后续脚本会统一通过 `Codes/accessor.py` 在分析阶段把它们从 nested schema 安全展开出来。

### 5.3 质量维度如何被操作化
- `complexity`
- `creativity`
- `domain_knowledge`
- `problem_solving`
- `real_world`
- `specificity`
- `technical_accuracy`

课堂口语版可以这样讲：

**“研究问题说的是‘长度偏好’，但持久化数据里先保存的是可复用原料；真正进入分析时，脚本再把这些原料翻译成 token 数、格式计数和任务布尔列。”**

---

## 6. 优化后的数据表意味着什么

优化后得到的是 `optimized_data.parquet`。  
它代表：

- 行数从约 13.6 万缩减到约 10.8 万
- 持久化顶层字段收紧为 14 列左右的 nested schema
- 长度、格式、任务标签等分析变量不再全部堆在顶层，而是按结构归类保存

更重要的是，它在工程上成为了后续 C14-C23 的统一输入接口。  
也就是说，前面的脚本是在“探索原始世界”，从这里开始，后面的脚本才是在“消费分析世界”。当前版本里，C14-C23 统一通过 `accessor` 共享层读取这些字段，而不再各自维护本地路径和字段转发函数。

这一步要让学生理解：

**样本变少了，不一定是坏事；如果少掉的是口径不一致、无法解释的记录，反而说明研究变严谨了。**

## 7. C13 为什么还要继续分子集

很多学生会问：

“既然已经有总表了，为什么不直接在全量数据上做检验？”

答案是：

**因为不同任务场景的人类偏好可能不同。**

举例：

- 创意写作里，长一点也许更像“展开描写”
- 数学里，长一点未必更好，可能还会显得啰嗦
- 代码任务里，格式和结构也可能比普通问答更重要

所以 C13 把数据分成两大类子集：

### 7.1 单类子集
只要包含某类任务就纳入，允许重叠。

它回答的问题是：

**“在所有代码任务里，偏好是什么样？”**

### 7.2 互斥纯净分区
精确枚举四个布尔标签的 True/False 组合，得到 `2^4 = 16` 个互不重叠的纯净子集。

它回答的问题是：

**“在只包含某种特定组合的样本中，偏好又是什么样？”**

再加上 4 个单类子集，总共形成 20 个子集文件。

还可以顺手提醒学生：后续脚本并不是每次都会把这 20 个文件全部用一遍。  
例如描述性脚本更常使用“全量 + 单类子集”，而 C16-C21 这类正式检验脚本更常使用“全量 + 16 个纯净分区”。  
这说明子集不是为了“看起来更复杂”，而是为了服务不同层次的问题。

## 8. 第二阶段最值得讲的统计前思想

这一阶段虽然还没有正式做显著性检验，但已经在训练学生两种重要思维：

### 思维 1：数据口径必须统一
如果同一个变量在不同记录里代表的含义不一样，就不能直接混在一起分析。

### 思维 2：总体结论和分层结论可能不同
同一个现象，在不同任务类型中可能强弱不同，甚至方向不同。

这也是为什么后续不仅有“全量分析”，还有大量“子集分析”。

---

## 9. 课堂展示时建议怎么讲

### 9.1 最推荐的讲解顺序

1. 先展示 C10 发现的 token 口径问题。
2. 再讲为什么只保留 `evaluation_order = 1`。
3. 然后展示 C12 产出的 nested schema 核心字段，以及后续脚本如何通过 `accessor` 在分析阶段展开临时列。
4. 最后展示 `R09_optimization_report.txt` 和 C13 的 20 个子集，告诉学生：后续所有统计检验都不是只在总表上做。

### 9.2 这一步最容易打动学生的点

不是“代码写得多漂亮”，而是：

**原来真正严谨的研究，不是先做复杂模型，而是先不断问：这个变量到底能不能这样解释？**

---

## 10. 本阶段结论

第二阶段完成后，项目已经从“原始复杂数据”进入“分析就绪数据”状态。  
学生在这一阶段应该真正掌握三件事：

1. 为什么原始数据不能直接分析。
2. 为什么 `evaluation_order = 1` 是关键过滤规则。
3. 为什么研究者要主动设计变量和子集，而不是被动接受原始字段。

如果这三点理解到位，学生就已经跨过了“会看字段名”和“能理解分析对象”之间最重要的一道坎。

学到这里，他们才真正准备好进入下一阶段：

**从图形观察走向统计检验。**
