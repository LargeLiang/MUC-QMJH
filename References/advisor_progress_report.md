## 一、课题定位与当前报告目的

本课题围绕 LMArena 人类偏好数据展开，核心关切并不是简单讨论“长回答是不是更好”，而是进一步追问：

**当人类在两个大语言模型输出之间做偏好选择时，是否会系统性地受到长度、标题、列表、粗体等形式特征影响；如果会，这种影响究竟是表层相关、混淆放大，还是在控制其他因素后仍然成立的独立效应。**

这一定义决定了本课题不是单一的可视化项目，也不是仅做一次显著性检验的统计练习，而是一条完整的研究链条：

1. 先确认原始数据结构是否可靠。
2. 再把复杂嵌套数据整理成可分析对象。
3. 然后通过图表和检验识别偏好现象。
4. 接着控制混淆变量并做稳健性验证。
5. 最后尝试用机制模型解释偏好是如何形成的。

本次过程报告的目标，是把现有成果、已有 Reference 文档和当前仓库状态统一整理成一份更适合导师把握全局的文本，既展示已经取得的结果，也如实说明当前工程整理状态与后续工作重点。

## 三、研究推进逻辑：从数据审计到机制解释

为了便于导师从整体上把握课题，我将现有脚本和文档统一归纳为五个阶段。

> 说明：导读文档 introduction_phase_5 中曾将 SEM 对应为 C21，但在后续脚本重编号后，当前仓库中 C21 为匹配诊断，C22 才是 SEM。以下以当前实际脚本编号为准。

### 3.1 第一阶段：确认研究对象可信

对应脚本：C01-C03  
对应导读：introduction_phase_1.md

这一阶段解决的不是“统计显著性”问题，而是“数据能不能被放心用于研究”的问题。

核心工作包括：

1. 检查同一 evaluation_session_id 是否跨多个 parquet 分片出现。
2. 将 7 个原始分片整合成统一分析入口。
3. 检查同一 Session 内 evaluation_order 是否连续、是否有重复。

这一阶段的重要意义在于，项目一开始就明确了数据的真实层级是 Session → Order → Turn，而不是把“一行数据”误认成“一个独立问答样本”。这一步为后续所有变量定义和清洗策略奠定了对象基础。

### 3.2 第二阶段：把复杂原始数据加工成可分析数据

对应脚本：C04-C13  
对应导读：introduction_phase_2.md

这一阶段是从“原始复杂对象”进入“统计分析数据表”的关键转折点。

主要工作包括：

1. 逐字段摸底：用 C04-C11 分别分析 session、order、turn、model、role、content、token、category_tag 等结构。
2. 发现关键口径问题：C10 识别出 sum_user_tokens 在 evaluation_order > 1 时混入整个 Session 历史，而不再只代表当前评价对象。
3. 清洗与过滤：C12 据此只保留 evaluation_order = 1，同时过滤空内容、异常语言和关键字段缺失记录。
4. 扁平化与特征工程：把嵌套对话结构整理为分析友好型列，并构造长度变量、格式变量、任务类型布尔列和质量维度列。
5. 子集划分：C13 将优化后的主表进一步拆分为 4 个单类子集与 16 个互斥纯净分区，共 20 个子集文件。

这一阶段的研究价值非常高，因为它真正完成了“研究变量的操作化”：

- 把“长度偏好”翻译为 token 差值；
- 把“格式偏好”翻译为标题数、列表数、粗体数与密度；
- 把“任务类型差异”翻译为可用于分层分析的布尔变量；
- 把“问题质量维度”翻译为可进入后续回归和机制模型的控制项。

### 3.3 第三阶段：从图表观察走向正式检验

对应脚本：C14-C17  
对应导读：introduction_phase_3.md

这一阶段开始正面回答“偏好现象是否存在”这一问题。

其中：

1. C14 对长度偏好进行描述性分析，围绕长度差值分箱、胜率变化与子集异质性展开。
2. C15 对格式偏好进行描述性分析，分别观察格式存在性、数量、组合与胜率的关系。
3. C16 使用 Wilcoxon 符号秩检验、Bonferroni 校正、Bootstrap 置信区间与效应量指标，验证长度偏好并非偶然波动。
4. C17 对标题、列表、粗体三类格式做配对检验，并额外引入密度检验，以控制长度混淆。

这一阶段的重要转变在于：

- 图表不再只是“看起来明显”；
- 偏好现象开始被转化为可检验、可量化、可比较的统计证据。

### 3.4 第四阶段：控制混淆并进行稳健性验证

对应脚本：C18-C21  
对应导读：introduction_phase_4.md

第三阶段只能说明现象存在，但还不能直接说“长度或格式本身导致偏好”。第四阶段的任务，就是把“相关”进一步推进到“净效应”和“稳健性”。

主要工作包括：

1. C18 嵌套逻辑回归：逐层加入任务类型、问题复杂度、模型能力与模型风格代理变量，观察长度和格式效应衰减情况。
2. C19 长度效应稳健性：在调整 Logit 的基础上结合稳定化 IPW，估计更长回复的平均处理效应。
3. C20 格式效应稳健性：对标题密度、粗体密度和列表密度分别做调整模型与 IPW 检验。
4. C21 匹配诊断：通过 1:1 最近邻匹配和 SMD 平衡性比较，验证在更可比样本中长度效应是否仍成立。

这一阶段的核心思想，是把“显著”进一步拆开问：

- 显著现象里有多少是混淆带来的？
- 扣除模型能力、任务类型等因素后还剩多少？
- 如果换一种估计方式，结论还稳不稳？

### 3.5 第五阶段：从净效应走向机制解释

对应脚本：C22  
对应导读：introduction_phase_5.md

在完成净效应和稳健性验证之后，课题进入最终机制解释阶段。C22 使用结构方程模型，把“模型差异”“回答长度”“格式密度”“最终偏好”放进同一张路径图中，尝试回答以下问题：

1. 长度效应是独立直接作用，还是某个更深层变量的代理？
2. 标题和粗体是直接提高偏好，还是通过别的中介起作用？
3. 模型能力与格式倾向会通过什么间接路径影响偏好判断？

这一阶段把整项研究从“现象识别”提升到了“机制理解”，也使论文叙事从“发现偏差”进一步上升为“解释偏差如何形成”。

## 四、阶段性研究结论与核心发现

### 4.1 数据基础已经被澄清并统一

根据 current_report 与各阶段导读，当前研究已经形成明确的数据基础认知：

- 原始数据规模为 135,634 条、14 列。
- C12 清洗后保留 108,171 条、32 列的优化数据。
- 过滤掉的 27,463 条记录中，最主要原因是 evaluation_order > 1 导致的 token 口径污染。
- 优化后的数据表已经能够稳定支撑长度、格式、任务类型、问题维度与模型差异的联合分析。

这意味着，课题最重要的一个方法论问题已经解决：后续所有统计不再建立在“口径不一致的数据对象”之上。

### 4.2 长度偏好具有稳定存在性

基于 current_report 中对 C14、C16、C18、C19、C21 的综合记录，目前关于长度偏好的结论已经较为完整：

1. 描述性层面，长度差值与胜率表现出很强正相关。
2. 全量样本中，获胜回复更长的比例约为 62.2%，中位差值约为 +125 tokens。
3. 在 13 个有效子集中，12 个子集在 Bonferroni 校正后仍显著。
4. 在加入任务类型、问题维度、模型能力与模型风格控制后，长度净效应仍保留，当前汇总结论为全量净 OR 约 1.355。
5. IPW 结果显示，更长的一侧平均可带来约 14.9 个百分点的获胜提升。
6. 匹配后 mean|SMD| 从 0.250 降到 0.060，ATE 仍约为 0.160，说明长度偏好不是简单的样本失衡假象。

因此，可以较为稳妥地向导师汇报：

**长度偏好不是偶然波动，也不是完全由模型能力差异制造出来的表面现象，而是在多种控制和稳健性检验后仍然成立的系统性效应。**

### 4.3 格式偏好存在，但内部机制并不相同

格式偏好的研究目前已经从“有无格式”推进到“哪类格式具有独立作用”的层面。

根据 C15、C17、C18、C20 的汇总结果：

1. 描述性层面，含任意格式的回复相比无格式回复，胜率大约提升 9.4 个百分点。
2. 其中粗体的描述性提升最明显，约为 +11.1%。
3. 正式检验显示，标题、列表、粗体三类格式在原始计数上都可能表现出显著性。
4. 但在进一步看格式密度后，列表密度不显著，说明它更多是长度效应的伴生指标。
5. 标题密度和粗体密度在控制长度和模型能力后仍保持正向独立效应。
6. 在格式稳健性分析中，标题密度调整 OR 约为 1.132，粗体密度调整 OR 约为 1.106，而列表密度不应进入主结论。

因此，本课题在格式偏好方面最值得强调的不是“格式越多越好”，而是：

**标题和粗体具有较稳定的独立格式效应，而列表更像被长度和模型风格共同裹挟的敏感性变量。**

### 4.4 混淆控制是本课题的重要方法进步

本课题的一项实质性提升，是没有停留在“做出显著结果”上，而是继续追问混淆来源。

M03_confounding_analysis 与 current_report 给出的核心认识是：

1. 模型能力差异是长度和格式分析中必须正视的最大混淆来源之一。
2. 任务类型、用户输入长度、对话轮数、问题质量维度等变量也会共同影响“应该生成多长、多结构化的回答”。
3. 在 C18-C22 中，这些混淆因素已经被系统纳入，而不是停留在口头承认层面。
4. 经过控制后，长度效应虽然明显衰减，但并未消失；格式效应则分化为“独立成立的标题/粗体”和“不应作为主结论的列表”。

这说明，本课题已经从“识别现象”推进到了“分解现象”，这一点对于后续论文写作和答辩都非常关键。

### 4.5 机制模型为整条研究链闭环

根据 current_report 中对 C22 的总结，当前 SEM 模型给出了较好的拟合指标：

- 主模型 CFI 约 0.9998
- 主模型 RMSEA 约 0.0030

更重要的是，路径层面的解释已经较为清晰：

1. 长度差具有显著直接效应。
2. 标题密度差和粗体密度差也具有显著直接效应。
3. 模型能力差异的影响并不完全通过长度通道传递，其通过标题和粗体的间接路径更稳定。
4. 长度与格式不是严格串联关系，更像是并列作用于偏好形成的平行通道。

这一点使本课题的最终表达不再只是“发现偏好偏差”，而可以进一步上升为：

**人类偏好数据中的形式偏差，不仅可以被识别和控制，还可以被机制化解释。**

## 五、研究过程中几个关键问题及其解决

### 5.1 原始数据结构复杂，研究单位容易误判

项目最初最大的困难之一，是原始 parquet 数据嵌套层级复杂，容易把一行数据误解成一个完整对话样本。通过 C01-C03 的会话完整性和 order 一致性验证，最终明确了一行数据对应的是一次评价记录，而不是一个完整 Session。这一步解决了后续所有分析对象定义问题。

### 5.2 token 口径污染问题直接影响长度研究有效性

如果没有 C10 的 token 校验，本课题最核心的长度变量会建立在错误对象上。通过发现 evaluation_order > 1 时 sum_user_tokens 混入整个 Session 历史，本项目最终把“只保留首轮评价”从一个经验性筛选，变成了有明确统计口径依据的关键过滤规则。

### 5.3 格式偏好存在“长度混淆”风险

早期如果仅观察格式计数，很容易得出“格式越多越容易赢”的表面结论。随后通过 C17 密度检验、C18 嵌套控制和 C20 稳健性分析，最终把标题/粗体和列表区分开，避免了把非独立效应误写成核心发现。

### 5.4 研究没有停留在相关分析

从 C18 到 C22 的推进，本质上是在不断提高研究主张的技术门槛：

- 从描述相关到配对检验；
- 从配对检验到控制混淆；
- 从控制混淆到加权与匹配稳健性；
- 从净效应到机制路径。

这使得整个课题形成了比较完整、层层递进的证据链，而不是单次统计显著的堆砌。

### 5.5 工程归档与输出规范也在同步推进

除统计分析外，近期还对仓库进行了较多规范化整理，包括：

1. 输出图表和 CSV 编号统一。
2. C14、C15 的命名规则对齐到成组编号格式。
3. 仓库治理、LICENSE 与公开前合规审查文档补充。
4. 数据结构说明、方法说明、教学导读等文档体系逐步建立。

这意味着本课题当前已经不只是“能跑出结果”，而是在向“可展示、可复现、可交付”的研究工程靠拢。

## 七、我在课题推进中的学习与能力提升

这一部分是本次过程报告中我希望主动向导师汇报的内容。与单纯完成若干脚本相比，我认为自己在以下几个方面的提升更重要。

### 7.1 从“会处理数据”提升到“先审计数据再做结论”

在课题早期，我更容易把注意力放在如何尽快做统计和画图上。但随着 C01-C03 和 C10 的推进，我逐渐真正理解：研究的第一步不是套方法，而是确认数据结构、研究单位和变量口径是否可信。

这一点对我的影响很大。现在我会先问：

- 一行数据到底代表什么？
- 变量在不同记录里是否含义一致？
- 若口径不一致，后面再复杂的模型是否都没有意义？

这是我在研究思维上的第一层提升。

### 7.2 从“做显著性检验”提升到“理解效应、混淆与稳健性”

通过 C16-C22 的推进，我对统计研究的理解明显超出了单纯的 p 值判断。

现在我已经能够较系统地区分：

1. 显著性是在回答“差异是否像随机噪声”。
2. 效应量是在回答“差异到底有多大”。
3. 混淆控制是在回答“这个差异是不是别的变量造成的”。
4. 稳健性分析是在回答“换一种方法结论还成不成立”。
5. 机制模型是在回答“这些变量通过什么路径影响结果”。

对我而言，这意味着自己开始真正进入“统计研究者”的训练，而不再只是“会调用统计函数”。

### 7.3 从“看现象”提升到“搭建证据链”

本课题对我最大的训练之一，是学会用多层证据支撑一个研究判断。

以前看到长度差值图和胜率曲线，我可能会较快认为“长度偏好显然存在”；但现在我会继续追问：

- 配对检验是否支持？
- 多重比较校正后是否仍成立？
- 控制模型能力后是否还成立？
- IPW、匹配诊断是否一致？
- SEM 是否能解释它在机制层面的地位？

这种“层层加码”的研究方式，让我认识到严谨研究依赖的是证据链，而不是单点结果。

### 7.4 从“写脚本”提升到“搭建研究工程”

随着课题推进，我对代码工程的认识也发生了变化。现在我更加重视：

- 脚本之间的职责边界是否清晰；
- 中间数据和结果文件命名是否规范；
- 图表、表格和报告之间是否能一一对应；
- 代码是否能够被复现、审阅和继续维护；
- 文档是否足以支撑对外展示和后续扩展。

近期对 C14、C15 输出编号、仓库规范文件、公开前合规审查和 Reference 文档的整理，实际上都在训练我从“完成任务”走向“交付工程”。

### 7.5 从“会做项目”提升到“能讲清项目”

在整理 introduction_phase_1-5、paper_framework、method_list、data_structure_guide 等文档的过程中，我明显感觉到自己对课题的理解更深了一层。因为只有当我能把研究逻辑讲清楚、能向零基础学生解释为什么这样设计、也能向导师解释每一步的必要性时，我才算真正吃透了这个项目。

因此，这一阶段我的另一个显著提升，是学术表达与研究叙事能力：

- 能把复杂工程压缩为清晰的阶段逻辑；
- 能把方法选择转化为可解释的研究语言；
- 能把代码、统计和论文叙事连接起来。

### 7.6 研究素养上的总体变化

如果用一句话总结我在本课题中的成长，我会概括为：

**我正在从“会做分析的人”逐步转向“能够围绕问题搭建完整研究链、并把结果规范呈现出来的人”。**

这既包括统计技术上的提升，也包括研究判断、工程组织和学术表达上的同步成长。
