# 第一阶段：认识数据——我们在研究什么？

> **受众**：无统计学背景的本科生  
> **目标**：理解研究问题的来源、数据集的结构，以及"清洗"数据意味着什么

---

## 1. 研究背景：人类为什么偏爱某些 AI 回答？

### 1.1 ChatGPT 之后的世界

你一定用过 ChatGPT、文心一言或 Kimi 等 AI 助手。当你同时看到两个模型对同一个问题的回答时，你会选哪个？**你的选择背后，有没有系统性的规律？**

这正是本研究要回答的问题。

大型语言模型（LLM, Large Language Model）的训练有一个关键环节叫做 **RLHF**（Reinforcement Learning from Human Feedback，基于人类反馈的强化学习）。简单来说，就是：

1. 让模型 A 和模型 B 各回答同一个问题
2. 请人类评估者选出更好的那个
3. 用这些偏好数据「教」模型，让它变得更符合人类期望

**问题来了**：人类的「喜好」真的反映了回答质量吗？还是说，人类其实是在被表面特征所影响——比如**回答更长**、**排版更漂亮**？

```
┌─────────────────────────────────────────────────────────┐
│  同一个问题                                             │
│                                                         │
│  模型 A 的回答：简洁，200字，纯文字                     │
│  模型 B 的回答：详细，800字，有标题、列表、粗体         │
│                                                         │
│  人类偏好 → 选 B                                        │
│                                                         │
│  问：是因为 B 质量更好？还是因为 B 看起来更"像样"？     │
└─────────────────────────────────────────────────────────┘
```

这个偏差，我们称为**长度偏好偏差**（length bias）和**格式偏好偏差**（format bias）。

---

### 1.2 这个偏差为什么重要？

- **它影响 AI 的进化方向**：如果 RLHF 数据充满"更长=更好"的偏见，训练出的模型会越来越倾向于用大量文字填充回答，而不是追求信息密度。
- **它在 AI 安全领域是已知问题**：学术界称之为「sycophancy」（谄媚），模型学会了 *取悦* 用户，而非 *帮助* 用户。
- **它是可以量化的**：这正是本研究的核心价值所在。

---

## 2. 数据集：LMArena

### 2.1 数据从哪里来？

本研究使用的是 **LMArena**（现称 Chatbot Arena）平台在 Hugging Face 上公开的数据集：

```
lmarena-ai / arena-human-preference-140k
```

这个平台的工作方式是：

```
用户提一个问题
    ↓
两个匿名模型同时给出回答（用户不知道是哪个模型）
    ↓
用户选择"哪个回答更好"，或点击"平局"/"两个都差"
    ↓
选择结果被记录为一条数据
```

> **为什么匿名很重要？** 如果用户知道是 GPT-4 回答的，可能会因为品牌效应而偏向它。匿名排除了这种干扰。

### 2.2 数据集的规模

| 项目 | 数值 |
|------|------|
| 原始记录数 | **135,634 条** |
| 涉及模型数 | **53 个**（含 GPT-4o、Claude、Gemini 等主流模型）|
| 数据存储格式 | Parquet（分 7 个文件分片）|
| 语言种类 | 126 种（英语约占 52.5%）|

### 2.3 Parquet 是什么？

普通人可能习惯用 Excel（.xlsx）或 CSV（.csv）存数据。**Parquet** 是一种专为大数据设计的文件格式，特点是：

- 按列压缩存储，文件体积比 CSV 小很多
- 读取速度极快，适合 Python/pandas 处理
- 支持复杂的嵌套数据结构（如列表里嵌列表）

```python
# 读取一个 CSV（小数据时常用）
df = pd.read_csv("data.csv")

# 读取一个 Parquet（大数据时更高效）
df = pd.read_parquet("data.parquet")
```

### 2.4 数据结构：一行记录代表什么？

原始数据共 **14 列**，其中最关键的字段如下：

| 字段名 | 中文含义 | 示例值 |
|--------|---------|--------|
| `evaluation_session_id` | 会话唯一标识符（UUID）| `b52ab86d-0f56-4fe6-8276-...` |
| `evaluation_order` | 本会话中的第几次评价 | `1`, `2`, `3`... |
| `model_a` / `model_b` | 参与对比的两个模型名称 | `gpt-4o`, `claude-3-5` |
| `winner` | 人类选择的胜者 | `model_a`, `model_b`, `tie`, `both_bad` |
| `conversation_a` / `conversation_b` | 两个模型的完整对话内容（列表嵌套字典）| 见下方说明 |
| `category_tag` | 问题所属的任务类别（字典格式）| `{"creative_writing_v0.1": {...}}` |
| `language` | 对话语言 | `English`, `Chinese` |

> **UUID** 是"通用唯一标识符"的缩写，形如 `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`，在计算机系统中用来唯一标识一条记录。

#### conversation 字段的嵌套结构

`conversation_a` 字段存储的不是普通文本，而是一个**列表套字典**的嵌套结构：

```python
[
    {"role": "user",      "content": "写一首关于秋天的诗"},
    {"role": "assistant", "content": "金风送爽叶飘零..."},
    {"role": "user",      "content": "可以改成更现代的风格吗？"},
    {"role": "assistant", "content": "秋天是一张旧照片..."},
]
```

每个字典代表一"轮"对话（一问一答）。这是多轮对话数据，也是我们后续需要"扁平化"处理的原因。

## 3. 数据验证阶段（C01–C03）

在正式分析之前，我们需要像医生检查病人档案一样——先验证数据是否完整、一致、没有结构性错误。这一步叫做**数据质量审计**。

### 3.1 C01：跨文件 Session 完整性验证

**问题**：原始数据被分成了 7 个文件（分片存储）。同一个会话的不同评价轮次，会不会被拆散到不同文件里？

**验证方法**：
- 读取每个文件，提取所有 `evaluation_session_id`（会话 ID）
- 对比：7 个文件的 ID 合并去重后，总数是否等于各文件独立去重之和？

```
文件1的唯一ID数：19,376  ┐
文件2的唯一ID数：19,404  │
文件3的唯一ID数：19,381  │  7个文件各自去重后相加
文件4的唯一ID数：19,376  ├─────────────────────────→ 总和 = 135,634
文件5的唯一ID数：19,405  │
文件6的唯一ID数：19,346  │
文件7的唯一ID数：19,346  ┘

全局去重后唯一ID数 → 也是 135,634（= 总和）

结论：✅ 没有跨文件重复，每条记录只出现在一个文件中
```

**结论**：7 个分片之间没有重叠，可以安全地合并。

> **为什么要做这个验证？**  
> 想象一下：同一个会话的第 1 轮在文件 1，第 2 轮在文件 2。如果拆开分析，每条记录看起来是独立的，但实际上它们属于同一次对话。这种结构性错误会导致分析结果严重失真。

---

### 3.2 C02：合并分片，生成整合数据

验证通过后，用 `pd.concat()` 将 7 个文件纵向拼接：

```python
# 伪代码示意
df_list = [pd.read_parquet(f) for f in 7个文件]
integrated_df = pd.concat(df_list, ignore_index=True)
# ignore_index=True 的作用：重新生成 0,1,2,3... 的行号，避免索引重复
```

**结果**：生成 `integrated_data.parquet`，共 **135,634 行 × 14 列**

---

### 3.3 C03：会话内部 Order 一致性验证

**新问题**：一个会话可以包含多轮评价（Order 1, 2, 3...）。我们需要确认：

1. 有没有某个会话的 Order 出现跳号（比如有 1 和 3，但缺 2）？
2. 有没有重复的 Order（同一会话里出现两个 Order=1）？

**验证方法**：按 `evaluation_session_id` 分组，统计每组的：
- 记录数量（count）
- 最大 Order 值（max_order）
- 唯一 Order 数（unique_order_count）

如果一切正常，三者之间应满足：`count = max_order = unique_order_count`

**实际结果**：

| evaluation_order 值 | 出现次数 |
|--------------------|---------|
| 1 | 108,315 行 |
| 2 | 15,972 行 |
| 3 | 5,350 行 |
| ... | ... |
| 28 | 1 行 |

> 绝大多数会话（75.5%）只有 1 轮评价；少数会话有 2–28 轮，说明有用户在同一次对话中进行了多次评价比较。

**关键发现**：Order > 1 的行（约 27,319 行）存在一个**严重问题**——

当一个会话进行到第 2、3 轮时，对话历史会累积。`sum_user_tokens`（用户输入的总 token 数）字段记录的是**整个会话历史的 token 总量**，而不只是当前这一轮。这意味着：如果我们用这个字段来分析"用户问题有多长"，Order > 1 的数据会给出虚假的"更长"的数值。

```
会话举例：
  第1轮：用户输入50个token → sum_user_tokens = 50  ✅ 正确
  第2轮：用户又输入30个token → sum_user_tokens = 80  ❌ 被历史累加
  第3轮：用户又输入20个token → sum_user_tokens = 100 ❌ 进一步累加
```

**解决方案**：在数据清洗阶段（C12），过滤掉所有 `evaluation_order > 1` 的行，只保留每个会话的第一轮评价，确保 token 数字段反映真实的单轮长度。

## 4. 数据清洗阶段（C12）

### 4.1 为什么需要清洗数据？

原始数据就像是未经处理的矿石——里面有我们需要的"黄金"，但也混杂着：

- **格式问题**：对话内容是嵌套字典，不能直接分析
- **质量问题**：部分记录的内容字段为空
- **结构问题**：Order > 1 的行有系统性 token 计数错误
- **标签问题**：任务类别以字典格式存储，需要转化为便于分析的布尔列

### 4.2 清洗规则

| 过滤条件 | 过滤行数 | 原因 |
|----------|---------|------|
| `evaluation_order > 1` | 27,319 行 | token 累积计数问题 |
| `conversation_a` 中存在空内容 | 55 行 | 数据残缺，无法分析 |
| `conversation_b` 中存在空内容 | 54 行 | 数据残缺，无法分析 |
| `language == "<err>"` | 30 行 | 语言识别失败，无法做语言分层分析 |
| 关键评分字段为 None | 5 行 | 标签缺失 |
| **合计** | **27,463 行（20.25%）** | |

**保留**：108,171 行（79.75%）

### 4.3 数据扁平化：把嵌套结构"展开"

原始的 `conversation_a` 字段是嵌套的列表+字典结构。我们把它"展平"，提取出每一轮的文本和关键指标：

```
原始（嵌套字典列表）:
  conversation_a = [
      {"role": "user",      "content": "xxx", "num_tokens": 15},
      {"role": "assistant", "content": "yyy", "num_tokens": 312},
  ]

展平后（直接列）:
  turn_1_a_text = "yyy"（第1轮模型A的回复文本）
  a_tokens = 312  （模型A的回复 token 数）
  a_header_count = 3  （回复中标题数量）
  a_list_count = 7    （回复中列表项数量）
  a_bold_count = 5    （回复中粗体标记数量）
```

### 4.4 最终字段说明（32列）

清洗后的 `optimized_data.parquet` 包含 32 列，核心字段如下：

| 列名 | 含义 |
|------|------|
| `id` | 行唯一标识 |
| `model_a` / `model_b` | 两个对比模型的名称 |
| `winner` | 胜者（`model_a`/`model_b`/`tie`/`both_bad`）|
| `a_tokens` / `b_tokens` | 两个模型各自回复的 token 数 |
| `a_header_count` / `b_header_count` | 回复中的标题数量（Markdown `#` 格式）|
| `a_list_count` / `b_list_count` | 回复中的列表项数量 |
| `a_bold_count` / `b_bold_count` | 回复中的粗体标记数量 |
| `creative_writing_bool` | 是否属于创意写作任务 |
| `if_bool` | 是否属于指令遵循任务 |
| `math_bool` | 是否属于数学任务 |
| `code_bool` | 是否属于代码任务 |

## 5. 本阶段关键概念小结

| 概念 | 一句话解释 |
|------|-----------|
| **Parquet** | 为大数据设计的压缩列式文件格式，比 CSV 更高效 |
| **UUID** | 全球唯一的字符串 ID，用来标识一条记录 |
| **数据分片（Sharding）** | 将大文件切分为多个小文件，便于存储和传输 |
| **会话（Session）** | 一次完整的人机对话过程，可包含多轮问答 |
| **评价轮次（Order）** | 同一会话中用户进行的第几次 A/B 比较 |
| **数据扁平化** | 将嵌套的字典/列表结构转化为可直接分析的表格列 |
| **Token** | 语言模型处理文本的基本单位，约等于"一个词"或"几个字符"；一个中文字约等于 2 token |
| **RLHF** | 基于人类反馈的强化学习，用人类偏好数据训练 AI |
