# 工程集中到单一 Notebook 的可行性评估

> 目标：探索是否可以把当前工程文件整合到一个适合课堂展示的 Jupyter Notebook 中  
> 当前结论：可以做，但应采用“脚本为主、Notebook 为教学外壳”的方案；目前已完成 Phase 1 集成

---

## 1. 先给结论

把整个工程“完全改写成一个大 Notebook”并不是最优方案。  
但把工程“包装成一个有叙事顺序的教学 Notebook”，是可行且很有价值的。

因此推荐的路线不是：

- 把所有 `.py` 脚本复制进 Notebook

而是：

- 保留 `.py` 脚本作为权威实现
- 用 Notebook 作为教学入口、运行入口和解释入口

这也是当前 `Codes/C00_all_collection.ipynb` 的定位。

---

## 2. 为什么 Notebook 对教学有明显优势

### 2.1 叙事连续
教师可以按“问题 -> 代码 -> 输出 -> 解释”的顺序展示，不必频繁切换多个文件。

### 2.2 低门槛
对初学者来说，Notebook 天然更像讲义，接受成本比一堆脚本低得多。

### 2.3 便于局部演示
可以只运行某一阶段，而不是每次都从头跑完整条分析链。

### 2.4 适合加入教学说明
Markdown 单元可以直接解释：

- 这段代码做什么
- 为什么这么做
- 输出该怎么理解

这正是课堂展示最需要的部分。

---

## 3. 为什么不建议把所有逻辑都搬进 Notebook

### 3.1 维护成本高
一旦脚本逻辑更新，Notebook 和 `.py` 版本很容易不同步。

### 3.2 差异追踪困难
Git 对 `.py` 文件更友好，对大 Notebook 的版本比较和审查更困难。

### 3.3 运行状态容易污染
Notebook 是有状态环境，单元运行顺序错误会带来隐藏问题。

### 3.4 重计算过重
像 C19-C21 这类脚本，已经带有 bootstrap 和模型拟合，如果全部塞进课堂 Notebook，运行时间和稳定性都会变差。

所以更合理的策略是：

**Notebook 负责“讲”和“调度”，脚本负责“算”和“产出”。**

---

## 4. 推荐的总集成方式

建议把单一 Notebook 设计成五个教学阶段，每个阶段都采用统一结构：

1. 先用 Markdown 解释研究问题
2. 再导入对应脚本函数
3. 再运行最核心的一步
4. 最后用 Markdown 总结输出意味着什么

这样做有两个好处：

- 学生能同时看到“代码”和“研究逻辑”
- 真正复杂的分析仍留在脚本里，Notebook 不会变成难维护的大杂烩

## 7. 课堂版与研究版应当区分

建议明确区分两种用途：

### 7.1 研究版
- 以 `.py` 脚本为主
- 追求可复现、可维护、可批处理

### 7.2 教学版
- 以 Notebook 为主
- 追求可解释、可展示、可分阶段运行

这两者不是竞争关系，而是互补关系。

---

## 8. 最推荐的工程组织原则

如果后续继续扩展 Notebook，建议始终遵守下面三条：

1. 核心分析逻辑只保留在 `.py` 中。
2. Notebook 只导入和调用，不复制核心实现。
3. Notebook 重点增加解释、预览、结果解读和教学提示。

这样才能既方便教学，又不破坏工程质量。

---

## 9. 结论

本项目完全可以做成“一个适合教学展示的总 Notebook”，但不应把它做成“唯一运行入口”。  
目前最合理的判断是：

- Phase 1 集成已经可用
- 全工程集成具有可行性
- 最优策略是“脚本主导、Notebook 讲解化包装”

这条路线既满足课堂展示，也不牺牲工程的可维护性。
